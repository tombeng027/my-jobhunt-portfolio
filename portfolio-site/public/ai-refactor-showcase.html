Legacy Field Mapper Refactor

An insurtech field-mapping module used to translate carrier policy payloads into an internal schema. Each new carrier required 150–200 lines of hand-written conditional mapping code — brittle, inconsistent, and prone to silent data errors. Using AI-assisted prompt engineering and automated test generation, I rebuilt it as a data-driven mapper that handles any carrier in ~40 lines.

← Back to Case Studies

The Problem

A legacy 1.0 configuration layer mapped carrier-specific policy payloads to an internal engine schema using long, nested if/else chains — one function per carrier, written by hand for every new client onboarded. Issues included:

  • New carrier mappings took 3–5 days of manual coding and review.
  • Silent failures: mismatched field names produced null values that passed validation but corrupted downstream claims data.
  • No unit tests — defects were caught in UAT or, worse, production.
  • Knowledge was locked in individual developers. No pattern, no reuse.

After ~40 lines · universal, data-driven 
// AI-refactored: data-driven mapper — works for any carrier
// New carrier = new entry in FIELD_MAP, not a new function

const FIELD_MAP = {
  first_name:  ['claimant.firstName',    'claimant.first_name'],
  last_name:   ['claimant.lastName',     'claimant.last_name'],
  dob:         ['claimant.dateOfBirth',  'claimant.dob'],
  policy_id:   ['policy.id',             'policy.policyNumber'],
  benefit_amt: ['claim.benefitAmount',   'claim.amount'],
  ssn:         ['claimant.ssn',          'claimant.taxId'],
  // Adding a new field: one line, done in minutes
};

function resolvePath(obj, path) {
  return path.split('.').reduce(
    (acc, key) => acc?.[key], obj
  ) ?? null;
}

function mapPayload(payload, fieldMap = FIELD_MAP) {
  return Object.fromEntries(
    Object.entries(fieldMap).map(([target, sources]) => [
      target,
      sources.reduce(
        (val, src) => val ?? resolvePath(payload, src),
        null
      ),
    ])
  );
}

// Usage — same call signature, any schema
const internal = mapPayload(carrierPayload);

Outcomes

~80% reduction in mapper code per carrier (190 → ~40 lines)
0 field-mapping incidents in 3 months post-refactor
38% faster carrier onboarding (4 months → 2.5 months)
100% legacy field knowledge preserved via FIELD_MAP config

How AI Was Used

  • Schema analysis: Fed the full set of carrier payload samples and the internal schema to an LLM. Prompted it to identify field name aliases and path patterns across all carriers.
  • Refactor generation: Used the LLM to propose a data-driven mapping strategy, then iterated on the implementation with targeted follow-up prompts for edge cases (nested paths, null coalescing).
  • Test generation: Prompted the LLM to generate a unit test suite for the new mapper covering field resolution, missing field fallback, and null-safety — ~30 test cases generated in one pass.
  • Documentation: Auto-generated inline JSDoc and a carrier onboarding guide from the FIELD_MAP schema definition.

The AI did not write the final code autonomously — it accelerated the design phase and generated test scaffolding that I reviewed, refined, and validated. Result: a one-day refactor that replaced two weeks of manual work.
